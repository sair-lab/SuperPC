# KITTI-360 Dataset Generation, Projection, and Training Pipeline

## Scope
This document explains:
- How KITTI-360 paired samples are generated (RGB + submap point cloud + metadata)
- How projection/backprojection should be done for rectified images
- How those artifacts are consumed by the training pipeline in this repository

The focus is camera-00 (image_00), but the same logic applies to image_01 by switching calibration keys.

## 1. Dataset Generation Summary
The external generation script (the one you shared) creates one folder per selected frame:

- output_root/<drive>/image_00/<frame>/
- <frame>_rgb.png
- <frame>_submap_xyzrgb.ply
- metadata.json

Core generation flow (from your script):
1. Load map points in world frame from static (and optional dynamic) tiles.
2. Load per-frame pose from cam0_to_world.txt.
3. Convert world points into camera frame using transform_world_to_camera.
4. Frustum filter in camera frame using intrinsics and image size.
5. FPS downsample to target count.
6. Save downsampled points and metadata.

Important: because points are transformed with transform_world_to_camera before saving, the saved .ply is expected to already be in camera coordinates (for that frame).

## 2. Pose Files and Coordinate Frames
Official KITTI-360 definitions:

- poses.txt:
  - 3x4 transform from GPU/IMU pose frame to world.
- cam0_to_world.txt:
  - 4x4 transform from rectified perspective camera-0 frame to world.

Practical implication:
- If you generate points from cam0_to_world.txt, your saved points are camera-frame points after world->camera conversion.
- If you generate points from poses.txt, you must chain camera extrinsics (calib_cam_to_pose.txt) to get camera transforms.

## 3. Rectified Projection Rules
Your RGB images are under data_rect, so they are rectified perspective images.

Use:
- perspective.txt
- P_rect_00 (or P_rect_01) for projection
- R_rect_00 (or R_rect_01) only if you need to rotate non-rectified camera points into rectified camera frame

Do not use distortion coefficients D_00/D_01 for data_rect projection.

## 4. Transform Chains (Reference)

### Case A: points already in camera frame (your generated submaps)
Project directly with P_rect_00.

### Case B: points in pose/GPU-IMU frame
Let:
- T_world_pose from poses.txt
- T_pose_cam0 from calib_cam_to_pose.txt
Then:
- T_world_cam0 = T_world_pose * T_pose_cam0
- T_cam0_world = inverse(T_world_cam0)
- x_cam0 = T_cam0_world * x_world
Then project with P_rect_00.

### Case C: points in world frame and cam0_to_world available
Let T_world_cam0 from cam0_to_world.txt:
- T_cam0_world = inverse(T_world_cam0)
- x_cam0 = T_cam0_world * x_world
Then project with P_rect_00.


## 5. Connection to Training Pipeline

### 5.1 Argument entry
- train_pufm.py accepts --dataset kitti360
- It dispatches to args/kitti360_args.py

### 5.2 Dataset loading
- dataset/dataset.py treats kitti360 as a map dataset path
- It scans records under kitti360_root
- For vision conditioning it returns image path and intrinsics from metadata when available

### 5.3 Imperfect input generation in training
Two options:
1. Online generation from GT .ply each epoch
2. Offline cache generated by preprocess_kitti360_imperfect_inputs.py and loaded with:
   - --use_preprocessed_input true

### 5.4 Vision conditioning path
- Dataset returns image tensor + intrinsics matrix
- Intrinsics are resized consistently with image resize
- Model consumes both in cross-attention projection

## 6. Current Repository Behavior to Keep in Mind
- The kitti360 preprocessing utility currently uses kitti360_num_occlusion_areas naming, while training args use num_occlusion_areas with compatibility fallbacks in dataset loader.
- For kitti360 eval split behavior, current implementation uses train/val split from available records (no dedicated scene-level test partition like Tartanair).

## 7. Practical Validation Checklist
When overlays look wrong:
1. Confirm image path is data_rect (rectified).
2. Confirm projection uses P_rect_00 (or P_rect_01).
3. Confirm whether points are camera-frame or pose/world-frame.
4. If camera-frame points are used, do not apply camera extrinsic transform again.
5. Compare projected coverage ratio (projected/in-point count) and visual overlay.
